# Mini-EDMS — Електронна система документообігу

Mini-EDMS — це навчальний прототип **системи електронного документообігу** з ролями, життєвим циклом документів, завантаженням файлів та повним журналом дій (Audit Trail).

Система орієнтована на внутрішній обіг документів у компанії (договора, акти, рахунки) з базовим погодженням та «електронним підписом» (імітація КЕП на рівні системи).

---

## 🎯 Цілі проєкту

- Показати **повний цикл роботи документа**: створення → погодження → підписання → архів.
- Продемонструвати **рольову модель доступу (RBAC)**.
- Реалізувати **зберігання файлів** та привʼязку їх до документів.
- Забезпечити **Audit Trail** для всіх змін.
- Надати простий, але наочний **веб‑інтерфейс** з різними сценаріями для ролей.

---

## 🔍 Основний функціонал

### Документи та життєвий цикл

Кожен документ має:

- Метадані:
  - `direction` — напрямок (вхідний, вихідний, внутрішній).
  - `title` — назва документа.
  - `type` — тип (договір, акт, рахунок тощо).
  - `counterparty` — контрагент (рядок).
  - `amount` — сума (число, опційно).
  - `dueDate` — дедлайн (опційно).
- Файли:
  - Один або кілька прикріплених файлів (PDF, зображення, DOCX, XLSX).
  - Зберігання шляху до файлу на сервері.
- Статус:
  - `draft` — чернетка, документ створений, ще не запущений у маршрут.
  - `on_approval` — документ на внутрішньому погодженні (перевіряє Керівник).
  - `on_signing` — погодження завершено, документ передано підписанту.
  - `signed` — документ підписано (фінальний робочий стан).
  - `rejected` — документ відхилено на етапі погодження або підписання.
  - `archived` — завершений документ перенесено в архів.

Дозволені переходи статусів:

- `draft` → `on_approval` (ініціатор відправляє).
- `on_approval` → `on_signing` (Керівник погоджує, перехід автоматичний).
- `on_approval` / `on_signing` → `rejected` (відхилення з коментарем).
- `on_signing` → `signed` (підписант накладає КЕП).
- `rejected` → `draft` (повернення на доопрацювання).
- `signed` → `archived` (архівація).

### Рольова модель (RBAC)

У системі передбачено 4 ролі:

1. **Головний Адміністратор (Super Admin)**
   - Унікальна роль, яка призначається лише одному користувачу (`admin@edms.local` за замовчуванням).
   - Має повний доступ до всіх функцій системи, включаючи керування іншими адміністраторами.
   - **Не може змінити свою роль** або бути видаленим/заблокованим іншими користувачами.

2. **Адміністратор (Admin)**
   - Керує користувачами (створення, редагування, блокування, призначення ролей), **окрім Головного Адміністратора та інших Адміністраторів**.
   - Бачить агреговану статистику по системі.
   - **Не має доступу до вмісту документів**, лише до метаданих і статистики (розмежування обовʼязків).
   - Керує довідниками (Відділи, Типи документів, Налаштування системи).
   - **Не може створювати або редагувати користувачів з роллю "Адміністратор"**.

3. **Керівник відділу (Approver)**
   - Бачить документи свого підрозділу.
   - Може:
     - Переглядати документи відділу в маршруті.
     - Погоджувати (`on_approval` → автоматично `on_signing`).
     - Відхиляти (`on_approval` → `rejected`) з обовʼязковим коментарем.
     - Переміщати в архів (`signed` → `archived`).

4. **Підписант (Signatory)**
   - Бачить документи зі статусом `on_signing` свого відділу.
   - Може накладати «електронний підпис» (`on_signing` → `signed`) або відхиляти.
   - Після підпису документ стає доступним лише для читання.

5. **Працівник (Employee)**
   - Ініціює документи:
     - Створення (`draft`).
     - Редагування та прикріплення файлів у статусах `draft` та `rejected`.
   - Бачить **лише свої документи**.
   - Може відправити документ у маршрут (`draft` → `on_approval`).

## 🧱 Структура бази даних (MongoDB)

### Колекція `users`

```js
User {
  _id: ObjectId,
  email: String,        // Унікальний логін
  passwordHash: String, // bcrypt
  fullName: String,
  role: String,         // 'admin' | 'approver' | 'signatory' | 'employee'
  department: String,   // Підрозділ користувача
  isBlocked: Boolean,   // Для тимчасового блокування
  createdAt: Date,
  updatedAt: Date
}
```

### Колекція `documents`

```js
Document {
  _id: ObjectId,
  title: String,
  direction: String,        // 'incoming' | 'outgoing' | 'internal'
  type: String,             // 'contract' | 'act' | 'invoice' | ...
  counterparty: String,     // Контрагент
  dueDate: Date,            // Опційно
  status: String,           // 'draft' | 'on_approval' | 'on_signing' | 'signed' | 'rejected' | 'archived'

  creator: ObjectId,        // ref: 'User' (Employee)
  approver: ObjectId,       // ref: 'User' (Approver), опційно
  signatory: ObjectId,      // ref: 'User' (Signatory), опційно

  files: [
    {
      _id: ObjectId,
      originalName: String,
      mimeType: String,
      size: Number,
      path: String,         // Шлях на сервері
      uploadedAt: Date
    }
  ],

  createdAt: Date,
  updatedAt: Date
}
```

### Колекція `auditLogs`

```js
AuditLog {
  _id: ObjectId,
  document: ObjectId,       // ref: 'Document'
  user: ObjectId,           // ref: 'User'
  action: String,           // 'create' | 'update' | 'status_change' | 'file_upload' | 'file_delete' | 'comment'
  fromStatus: String,       // Опційно (для переходів)
  toStatus: String,         // Опційно (для переходів)
  comment: String,          // Опційний коментар (наприклад, причина відхилення)
  createdAt: Date
}
```

## 🌐 REST API — ендпоінти

### Автентифікація

`POST /api/auth/login`
- Вхід: `{ email, password }`
- Вихід: `{ token, user: { id, fullName, role } }`
- Використовується JWT.

`GET /api/auth/me`
- Повертає поточного користувача за токеном.

---

### Користувачі (тільки Admin)

`GET /api/users`
- Отримати список користувачів.

`POST /api/users`
- Створити користувача.
- Вхід: `{ email, password, fullName, role, department }`.

`PATCH /api/users/:id`
- Оновити користувача (імʼя, роль, блокування).

`DELETE /api/users/:id`
- Видалити користувача.

---

### Документи

`GET /api/documents`
- Для `employee`: повертає лише документи, де `creator == currentUser`.
- Для `approver`: документи свого підрозділу.
- Для `signatory`: документи свого підрозділу в статусі `on_signing`.
- Для `admin`: тільки метадані (без детального вмісту/файлів, якщо хочеш підкреслити безпеку).

`GET /api/documents/:id`
- Повернути документ, якщо користувач має до нього доступ (RBAC‑перевірка).

`POST /api/documents`
- Створити документ у статусі `draft`.
- Доступно лише ролі `employee`.
- Вхід: `{ title, type, counterparty, dueDate? }`.

`PUT /api/documents/:id`
- Оновити документ (метадані).
- Доступно:
  - `employee`, якщо він creator і документ у статусі `draft` або `rejected`.
- Не дозволяється змінювати `status` напряму (тільки через окремі переходи).

`DELETE /api/documents/:id`
- Видалити документ:
  - Тільки якщо `status === 'draft'` і користувач — creator.
  - Або адмін (як адміністративна операція, якщо потрібно).

---

### Переходи статусів

`POST /api/documents/:id/submit`
- `draft` → `on_approval`.
- Тільки creator (`employee`).

`POST /api/documents/:id/approve`
- `on_approval` → `on_signing` (автоматично).
- Тільки `approver`.

`POST /api/documents/:id/reject`
- `on_approval` / `on_signing` → `rejected`.
- Тільки `approver` або `signatory`.
- Вхід: `{ comment }` (обовʼязковий).
- Коментар зберігається в `auditLogs`.

`POST /api/documents/:id/sign`
- `on_signing` → `signed`.
- Тільки `signatory`, який закріплений за документом.

`POST /api/documents/:id/archive`
- `signed` → `archived`.
- `approver` або `admin`.

---

### Файли

`POST /api/documents/:id/files`
- Додавання файлу до документа.
- `multipart/form-data`, поле `file`.
- Доступно:
  - Для `employee` — якщо документ у `draft` або `rejected`.
  - Для інших ролей за потреби (можна обмежити).

`DELETE /api/documents/:id/files/:fileId`
- Видалити файл.
- Аналогічно — тільки в `draft`/`rejected` і тільки creator або адмін.

---

### Audit Trail

`GET /api/documents/:id/audit`
- Повертає список `auditLogs` по документу.
- Доступ для всіх, хто має доступ до документа.

---

### Аналітика / Дашборд

`GET /api/stats`
- Повертає агреговану статистику для дашборду.
- Для `employee`: рахує лише їхні власні документи.
- Для `signatory`: рахує лише документи, що очікують підпису.
- Для `manager` та `admin`: загальна статистика.

## 🧮 Технологічний стек

- **Backend:** Node.js, Express.js
- **База даних:** MongoDB (Mongoose)
- **Frontend:** HTML5 + Vanilla JS + Tailwind CSS (або простий CSS)
- **Auth:** JWT, bcryptjs
- **Файли:** Multer (зберігання на диск)

## 🖥️ Фронтенд: вигляд для кожної ролі

### Спільні речі

- Верхня панель:
  - Назва системи.
  - Імʼя користувача + роль.
  - Кнопка «Вийти».
- Ліве меню:
  - Посилання на основні сторінки: «Мої документи» / «Реєстр документів» / «Дашборд» / «Користувачі» (для admin).

---

### Employee (Працівник)

Сторінки:

1. **Dashboard (My Overview)**
   - Лічильники:
     - `Draft`, `Pending`, `Approved`, `Signed`, `Rejected`, `Archived`.
   - Список останніх 5–10 документів.

2. **Мої документи (My Documents)**
   - Таблиця:
     - Назва, Тип, Контрагент, Статус, Дедлайн, Останнє оновлення, Дії.
   - Дії:
     - Відкрити.
     - Редагувати (лише `draft`/`rejected`).
     - Видалити (лише `draft`).

3. **Створення / Редагування документа**
   - Форма:
     - Назва, Тип (select), Контрагент, Сума, Дедлайн.
     - Блок завантаження файлів.
   - Кнопки:
     - «Зберегти як чернетку».
     - «Відправити на розгляд» (перехід `draft` → `on_approval`).

4. **Перегляд документа**
   - Метадані (read only).
   - Список файлів з превʼю (PDF відкривається у `<iframe>`, зображення — `<img>`).
   - Стрічка Audit Trail (таблиця/список).
   - Якщо статус `rejected` — відображається причина відхилення.

---

### Approver (Керівник)

Сторінки:

1. **Dashboard**
   - Лічильники за статусами по підрозділу/всім документам.
   - Блок «Документи, що очікують розгляду» (status = `on_approval`).
   - Блок «Підписані, але не в архіві» (status = `signed`).

2. **Реєстр документів**
   - Таблиця з фільтрами:
     - Фільтр по статусу, типу, контрагенту, ініціатору.
   - Дія «Відкрити» документ.

3. **Перегляд документа**
   - Всі метадані та файли.
   - Кнопки:
     - Для `on_approval`: «Погодити» (→ `on_signing`), «Відхилити» (→ `rejected` з модалкою «причина»).
     - Для `signed`: «В архів» (→ `archived`).

4. **Audit Trail**
   - На сторінці документа — стрічка всіх дій.

---

### Signatory

Сторінки:

1. **Dashboard**
   - Лічильник «Документи, що чекають мого підпису».
   - Список документів в статусі `on_signing` для відділу підписанта.

2. **Список документів на підпис**
   - Таблиця:
     - Назва, Тип, Контрагент, Статус, Дедлайн.
   - Дія «Відкрити».

3. **Перегляд документа**
   - Метадані, файли (перевірка змісту).
   - Кнопка:
     - «Підписати» (→ `signed`), «Відхилити».
   - Після підпису кнопка зникає, статус змінюється.

---

### Admin

Сторінки:

1. **Системний дашборд**
   - Загальна статистика:
     - К-сть користувачів за ролями.
     - К-сть документів за статусами.
     - Графік «документи по днях» (опційно для краси).

2. **Користувачі**
   - Таблиця:
     - Імʼя, email, роль, статус (активний/заблокований).
   - Дії:
     - Створити нового користувача (форма).
     - Змінити роль.
     - Заблокувати/розблокувати.
     - Видалити.

3. **Реєстр документів (оглядовий)**
   - Таблиця всіх документів (без можливості змінювати контент; опційно — з можливістю архівації чи адміністративного видалення).

## 📦 Встановлення та запуск

### 1. Клонування та залежності

```bash
git clone <ваш-репозиторій>
cd mini-edms
npm install
```

### 2. Змінні оточення

Створіть `.env` у корені проєкту:

```env
PORT=5001
MONGODB_URI=mongodb+srv://<username>:<password>@cluster.mongodb.net/mini-edms
JWT_SECRET=your_super_secret_key_here
UPLOAD_DIR=uploads
```

### 3. Ініціалізація бази (seed)

```bash
node seed.js
```

Скрипт створює:

- Адміністратора: `admin@edms.local`
- Керівника: `approver@edms.local`
- Підписанта: `signatory@edms.local`
- Співробітника: `employee@edms.local`
- Пароль для всіх: `password123`

### 4. Запуск

```bash
npm run dev
```

- Backend: `http://localhost:5001`
- Frontend: `http://localhost:3000`

## 🔐 Безпека та обмеження

- JWT‑автентифікація, паролі зберігаються як bcrypt‑хеші.
- RBAC перевіряється на рівні middleware для кожного ендпоінту.
- Завантаження файлів:
  - Ліміт розміру (напр. 10 МБ).
  - Білий список MIME‑типів.
- Логування всіх критичних дій в `AuditLog`.
