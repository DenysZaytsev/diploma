# Mini-EDMS — Електронна система документообігу

Mini-EDMS — це навчальний прототип **системи електронного документообігу** з ролями, життєвим циклом документів, завантаженням файлів та повним журналом дій (Audit Trail).

Система орієнтована на внутрішній обіг документів у компанії (договора, акти, рахунки) з базовим погодженням та «електронним підписом» (імітація КЕП на рівні системи).

---

## Цілі проєкту

- Показати **повний цикл роботи документа**: створення -> погодження -> підписання -> архів.
- Продемонструвати **рольову модель доступу (RBAC)**.
- Реалізувати **зберігання файлів** з версіюванням та привʼязку їх до документів.
- Забезпечити **Audit Trail** для всіх змін.
- Реалізувати **систему сповіщень** (email + in-app).
- Надати простий, але наочний **веб-інтерфейс** з різними сценаріями для ролей.

## Основний функціонал

### Документи та життєвий цикл

Кожен документ має:

- Метадані:
  - `direction` — напрямок (вхідний, вихідний, внутрішній).
  - `title` — назва документа.
  - `type` — тип (з динамічного довідника: договір, акт, рахунок тощо).
  - `counterparty` — контрагент (рядок).
  - `dueDate` — дедлайн (опційно).
  - `confidentiality` — гриф конфіденційності (`public` / `internal` / `confidential` / `secret`).
  - `tags` — масив тегів/ключових слів для категоризації та пошуку.
- Файли:
  - Один або кілька прикріплених файлів (PDF, зображення, DOCX, XLSX).
  - Зберігання шляху до файлу на сервері.
  - **Версіювання файлів** — при заміні файлу попередня версія зберігається в історії (`fileVersions`).
  - Автоматична фіксація: хто та коли завантажив файл (`uploadedBy`).
- Статус:
  - `draft` — чернетка, документ створений, ще не запущений у маршрут.
  - `on_approval` — на погодженні (перевіряє Керівник).
  - `on_signing` — погодження завершено, документ передано підписанту.
  - `signed` — документ підписано (фінальний робочий стан).
  - `rejected` — документ відхилено на етапі погодження або підписання.
  - `archived` — завершений документ перенесено в архів.
- Звʼязки:
  - **Повʼязані документи** — двонаправлені звʼязки між документами (`relatedDocuments`).

Дозволені переходи статусів:

- `draft` -> `on_approval` (ініціатор відправляє).
- `on_approval` -> `on_signing` (Керівник погоджує, перехід автоматичний).
- `on_approval` / `on_signing` -> `rejected` (відхилення з коментарем).
- `on_signing` -> `signed` (підписант накладає КЕП).
- `rejected` -> `draft` (документ повертається ініціатору на доопрацювання, де його можна редагувати та відправити повторно).
- `signed` -> `archived` (архівація).

### Рольова модель (RBAC)

У системі передбачено 5 ролей:

1. **Головний Адміністратор (Super Admin)**
   - Унікальна роль, яка призначається користувачу (наприклад, `oleksandr-it@gmail.com`) і має прапорець `isSuperAdmin: true`.
   - Має повний доступ до всіх функцій системи, включаючи керування іншими адміністраторами.
   - **Єдиний, хто може створювати нових користувачів з роллю «Адміністратор»**.
   - **Не може змінити свою роль** або бути видаленим/заблокованим.

2. **Адміністратор (Admin)**
   - Керує користувачами (створення, редагування, блокування, призначення ролей), **окрім Головного Адміністратора та інших Адміністраторів**.
   - **Не може призначати роль «Адміністратор»** новим або існуючим користувачам.
   - Керує довідниками: **Відділи**, **Типи документів**.
   - Керує **загальними налаштуваннями** системи (напр. ліміт на завантаження файлів).
   - Бачить агреговану статистику по системі та **журнал дій інших адміністраторів**.
   - **Повністю ізольований від вмісту документів** (не має доступу до сторінки перегляду) — бачить лише загальну статистику системи (розмежування обовʼязків).

3. **Керівник відділу (Approver)**
   - Може переглядати документи всіх відділів (прозорий реєстр), але може здійснювати дії (погодження, відхилення, архівація) **лише для документів свого підрозділу**.
   - Може:
     - Переглядати документи в реєстрі.
     - Погоджувати (`on_approval` -> `on_signing`).
     - Відхиляти (`on_approval` -> `rejected`) з обовʼязковим коментарем.
     - Переміщати в архів (`signed` -> `archived`).
     - Залишати коментарі до документів свого відділу.
   - Може **делегувати повноваження** іншому користувачу на визначений період.

4. **Підписант (Signatory)**
   - Може переглядати документи свого відділу зі статусом `on_signing` у прозорому реєстрі.
   - Може накладати «електронний підпис» (`on_signing` -> `signed`) або відхиляти **лише для документів свого відділу**.
   - Після підпису документ стає доступним лише для читання.
   - Може **делегувати повноваження** іншому користувачу на визначений період.

5. **Працівник (Employee)**
   - Ініціює документи:
     - Створення (`draft`).
     - Редагування та прикріплення файлів у статусах `draft` та `rejected`.
   - Бачить **лише свої документи**.
   - Може відправити документ у маршрут (`draft` -> `on_approval`).
   - Може виконувати **масові дії** (групове відправлення / видалення документів).

### Делегування повноважень

Керівники та підписанти можуть делегувати свої повноваження іншому користувачу:

- Вказуються дати початку та закінчення делегування.
- Делегат автоматично отримує право погоджувати/підписувати документи відповідного відділу.
- Система перевіряє активні делегування при кожній дії.
- Підтримка причини делегування (наприклад, «відпустка»).

### Email-сповіщення

Система автоматично надсилає email-сповіщення через SMTP (Nodemailer):

- **Керівникам** — при надходженні нового документа на погодження.
- **Підписантам** — при передачі документа на підписання.
- **Ініціаторам** — при зміні статусу документа (відхилено, підписано, архівовано).
- **Користувачам** — при зміні пароля або email адміністратором.

Кожен користувач може налаштувати свої сповіщення в **профілі** (увімкнути/вимкнути окремі категорії).

### In-app сповіщення

Окрім email, система має **внутрішні сповіщення** (дзвіночок у хедері):

- Лічильник непрочитаних сповіщень.
- Випадне меню зі списком сповіщень.
- Типи: `status_change`, `new_task`, `comment`, `deadline`, `delegation`, `system`.
- Позначення як прочитане (одне або всі одразу).
- Клік по сповіщенню веде до відповідного документа.

### Профіль користувача

Кожен користувач має доступ до налаштувань свого профілю:

- Завантаження аватару.
- Налаштування email-сповіщень за категоріями (для адмінів — системні попередження та дії з користувачами; для інших — нові завдання та зміни статусів).

### Повнотекстовий пошук

Система підтримує **пошук по вмісту файлів** (PDF, DOCX, TXT, CSV, MD):

- При завантаженні файлів текст автоматично витягується у фоновому режимі.
- Підтримувані формати: PDF (через `pdf-parse`), DOCX (через `mammoth`), текстові файли.
- Витягнутий текст зберігається в полі `textContent` (з MongoDB text index).
- Пошук відбувається як по метаданих, так і по вмісту файлів.

### Збережені фільтри

Користувачі можуть зберігати часто використовувані комбінації фільтрів у реєстрі:

- Зберігання стану всіх фільтрів (тип, статус, відділ, напрямок, теги, конфіденційність, дати тощо).
- Швидке застосування збережених фільтрів одним кліком.
- До 20 збережених фільтрів на користувача.

### Масові операції

Працівники можуть виконувати групові дії над кількома документами:

- Множинний вибір документів у реєстрі (чекбокси).
- Масове відправлення на розгляд (draft -> on_approval).
- Масове видалення чернеток.
- Обробка до 50 документів за одну операцію.

### Експорт даних

- **CSV-експорт реєстру документів** — завантаження списку документів з усіма метаданими у форматі CSV (включаючи теги та конфіденційність).
- **CSV-експорт журналу аудиту** — завантаження логів системного аудиту у CSV.
- **JSON-експорт паспорта документа** — для архівованих документів можна завантажити повний паспорт у форматі JSON.

## Структура бази даних (MongoDB)

### Колекція `users`

```js
User {
  _id: ObjectId,
  email: String,        // Унікальний логін
  passwordHash: String, // bcrypt
  fullName: String,
  role: String,         // 'admin' | 'approver' | 'signatory' | 'employee',
  department: String,   // Назва відділу (зв'язок за назвою)
  isBlocked: Boolean,   // Для тимчасового блокування
  isSuperAdmin: Boolean,// Прапорець Головного Адміністратора
  avatar: String,       // Шлях до аватару
  notifications: Object,// Налаштування email-сповіщень
  pendingEmail: String, // Email, що очікує підтвердження
  emailConfirmationToken: String,
  createdAt: Date,
  updatedAt: Date
}
```

### Колекція `documents`

```js
Document {
  _id: ObjectId,
  regNumber: String,        // Реєстраційний номер (унікальний, crypto.randomInt з retry)
  title: String,
  direction: String,        // 'incoming' | 'outgoing' | 'internal'
  type: String,             // Код типу з довідника 'documentTypes'
  department: String,       // Назва відділу (належить відділу ініціатора)
  counterparty: String,     // Контрагент
  dueDate: Date,            // Опційно
  status: String,           // 'draft' | 'on_approval' | 'on_signing' | 'signed' | 'rejected' | 'archived'

  creator: ObjectId,        // ref: 'User' (Employee)
  approver: ObjectId,       // ref: 'User' (Approver), призначається при погодженні
  signatory: ObjectId,      // ref: 'User' (Signatory), призначається при підписанні

  files: [fileVersionSchema],   // Поточні файли (version, uploadedBy)
  fileVersions: [{              // Історія замінених файлів
    fileId, originalName, mimeType, size, path, version, uploadedAt, uploadedBy, replacedAt
  }],

  tags: [String],               // Теги/ключові слова
  relatedDocuments: [ObjectId], // Зв'язки з іншими документами (двонаправлені)
  confidentiality: String,      // 'public' | 'internal' | 'confidential' | 'secret'
  textContent: String,          // Витягнутий текст файлів (select: false, text index)

  isDeleted: Boolean,       // Soft delete
  createdAt: Date,
  updatedAt: Date
}
```

## REST API — ендпоінти

### Автентифікація

`POST /api/auth/login`
- Вхід: `{ email, password }`
- Вихід: `{ _id, fullName, email, role, department, token }`
- Використовується JWT.
- Заблоковані користувачі не можуть увійти (повертається 403).
- **Rate limiting:** максимум 15 запитів за 15 хвилин.

`POST /api/auth/register`
- Публічна реєстрація (для прототипування).
- Роль завжди призначається `employee` (незалежно від вхідних даних).
- Вхід: `{ fullName, email, password, department }`
- **Rate limiting:** максимум 15 запитів за 15 хвилин.

`GET /api/auth/profile`
- Повертає поточного користувача за токеном.

`PATCH /api/auth/profile`
- Оновлення профілю (аватар, налаштування сповіщень).
- Приймає `multipart/form-data` з полем `avatar` та JSON-рядком `notifications`.

---

### Користувачі (тільки Admin)

`GET /api/users`
- Отримати список користувачів.
- Підтримує фільтри: `?role=...&department=...`

`POST /api/users`
- Створити користувача.
- Вхід: `{ email, password, fullName, role, department }`.
- Звичайний адмін не може створювати роль `admin` (тільки Super Admin).

`PATCH /api/users/:id`
- Оновити користувача (імʼя, роль, email, відділ, блокування, пароль).
- Зміна email потребує підтвердження через посилання на пошту.
- Зміна пароля надсилає новий пароль на email користувача.
- Обмеження: захист Super Admin від редагування/видалення; звичайний адмін не може редагувати інших адмінів.

`DELETE /api/users/:id`
- Видалити користувача.
- Не можна видалити себе, Super Admin або (для звичайного адміна) іншого адміна.

---

### Системний аудит (тільки Admin)

`GET /api/users/system/audit`
- Отримати журнал системних подій.
- Підтримує фільтри: `?search=...&action=...&dateFrom=...&dateTo=...&sortBy=...&sortOrder=...`
- Поле `sortBy` обмежене білим списком: `createdAt`, `action`, `adminName`, `adminEmail`, `targetEmail`.

`POST /api/users/system/audit/login`
- Логування входу в систему (викликається з фронтенду після успішного логіну).

`POST /api/users/system/audit/clear-request`
- Запит на очищення журналу аудиту (надсилає email з підтвердженням).

`GET /api/users/system/audit/clear-confirm/:token`
- Підтвердження очищення журналу аудиту за токеном з email.

### Підтвердження email

`GET /api/users/confirm-email/:token`
- Підтвердження зміни email-адреси за токеном.

### Документи

`GET /api/documents`
- Для `employee`: повертає лише документи, де `creator == currentUser`.
- Для `approver`, `signatory`: прозорий реєстр — бачать документи всіх відділів. Дії обмежені лише до свого відділу.
- Для `admin`: адмін не має доступу до перегляду документів (ізоляція від вмісту).
- Підтримує фільтри: `?type=...&status=...&search=...&direction=...&department=...&deadlineBefore=...&createdFrom=...&createdTo=...&tags=...&confidentiality=...`
- Документи з грифом `secret` видимі лише для користувачів того ж відділу.
- Пошук відбувається по метаданих та по вмісту файлів (повнотекстовий пошук).
- Пошукові запити санітизуються для захисту від ReDoS.

`GET /api/documents/:id`
- Повернути документ, якщо користувач має до нього доступ (RBAC-перевірка).
- Employee бачить лише свої документи.

`POST /api/documents`
- Створити документ у статусі `draft`.
- Доступно лише ролі `employee`.
- Вхід: `{ title, direction, type, counterparty, dueDate?, confidentiality?, tags? }` + опційно файли через `multipart/form-data`.
- Реєстраційний номер генерується автоматично з використанням `crypto.randomInt()` з механізмом повтору для уникнення колізій.
- Текст файлів витягується у фоновому режимі для повнотекстового пошуку.

`PATCH /api/documents/:id`
- Оновити документ (назва, контрагент, дедлайн, конфіденційність, теги).
- Доступно: `employee`, якщо він creator і документ у статусі `draft` або `rejected`.
- Не дозволяється змінювати `status` напряму (тільки через окремі переходи).

`DELETE /api/documents/:id`
- Soft delete документа.
- Доступно: `employee` (лише чернетки) або `admin` (будь-який статус).

### Масові операції

`POST /api/documents/bulk`
- Масова дія над кількома документами.
- Вхід: `{ action: 'submit' | 'delete', documentIds: [ObjectId] }`.
- Максимум 50 документів за раз.
- Повертає результат для кожного документа (успіх/помилка).
- Доступно: `employee` (лише свої документи).

### Переходи статусів

`POST /api/documents/:id/submit`
- `draft` / `rejected` -> `on_approval`.
- Тільки creator (`employee`).
- Надсилає email та in-app сповіщення керівникам відділу.
- Перевіряє активні делегування для пошуку заступника.

`POST /api/documents/:id/approve`
- `on_approval` -> `on_signing` (автоматично).
- Тільки `approver` свого відділу або активний делегат.
- Надсилає email та in-app сповіщення підписантам відділу та ініціатору.

`POST /api/documents/:id/reject`
- `on_approval` / `on_signing` -> `rejected`.
- Тільки `approver` або `signatory` (кожен лише свого відділу або через делегування).
- Вхід: `{ comment }` (обовʼязковий).
- Коментар зберігається в `auditLogs`.
- Надсилає email та in-app сповіщення ініціатору.

`POST /api/documents/:id/sign`
- `on_signing` -> `signed`.
- Тільки `signatory` свого відділу або активний делегат.
- Надсилає email та in-app сповіщення ініціатору та керівнику.

`POST /api/documents/:id/archive`
- `signed` -> `archived`.
- Тільки `approver`.
- Надсилає in-app сповіщення ініціатору.

---

### Коментарі

`POST /api/documents/:id/comments`
- Додати коментар до документа.
- Вхід: `{ comment }`.
- Доступно: employee (свої документи), approver та signatory (документи свого відділу).
- Надсилає in-app сповіщення ініціатору документа.

### Файли

`POST /api/documents/:id/files`
- Додавання файлів до документа.
- `multipart/form-data`, поле `files` (підтримує множинне завантаження, до 30 файлів за раз).
- Доступно: для `employee` — якщо документ у `draft` або `rejected` і користувач є creator.
- Ліміт загальної кількості файлів налаштовується в системних налаштуваннях.
- Нові файли отримують `version: 1` та `uploadedBy`.
- Текст автоматично витягується для повнотекстового пошуку.

`PUT /api/documents/:id/files/:fileId`
- **Заміна файлу** з збереженням попередньої версії в `fileVersions`.
- Новий файл отримує інкрементований номер версії.
- Доступно: `employee` (свої документи, статус `draft` або `rejected`).

`DELETE /api/documents/:id/files/:fileId`
- Видалити файл.
- Доступно: employee (свої документи) у статусі `draft`/`rejected`.

`POST /api/documents/:id/related`
- Додати звʼязок з іншим документом.
- Вхід: `{ relatedDocumentId }`.
- Створює **двонаправлений** звʼязок (обидва документи отримують посилання один на одного).

`DELETE /api/documents/:id/related/:relatedId`
- Видалити звʼязок між документами.
- Також двонаправлене видалення.

## Технологічний стек

- **Backend:** Node.js, Express.js v5
- **База даних:** MongoDB (Mongoose v9)
- **Frontend:** HTML5 + Vanilla JS + Tailwind CSS (CDN)
- **Auth:** JWT (30-денний термін дії), bcryptjs
- **Файли:** Multer (зберігання на диск, перевірка розширень)
- **Email:** Nodemailer (SMTP)
- **Безпека:** helmet (HTTP security headers), express-rate-limit
- **Повнотекстовий пошук:** pdf-parse (PDF), mammoth (DOCX)

## Безпека

- **Helmet** — HTTP security headers (CSP, X-Content-Type-Options, X-Frame-Options тощо).
- **Rate limiting** — обмеження 15 запитів за 15 хвилин на endpoints логіну та реєстрації (express-rate-limit).
- **CORS** — конфігурований список дозволених origins через змінну оточення `CORS_ORIGINS`.
- **JWT-автентифікація** (термін дії 30 днів), паролі зберігаються як bcrypt-хеші.
- **Блокування користувачів** — заблоковані користувачі не можуть увійти в систему та отримують помилку 403 при будь-яких API-запитах.
- **RBAC** перевіряється на рівні middleware (`protect` + `authorize`) для кожного ендпоінту.
- **Захист від XSS** — всі дані користувача екрануються через `escapeHtml()` перед вставкою в DOM.
- **Захист від ReDoS/NoSQL Injection** — пошукові запити санітизуються через `escapeRegex()` перед використанням в MongoDB `$regex`.
- **Конфіденційність документів** — документи з грифом `secret` видимі лише для користувачів того ж відділу.
- **Завантаження файлів:**
  - Ліміт розміру: 10 МБ.
  - Білий список розширень файлів: PDF, DOC, DOCX, XLS, XLSX, JPEG, JPG, PNG, GIF.
  - Конфігурований ліміт кількості файлів.
- **Білий список sortBy** — поле сортування для системного аудиту обмежене дозволеними значеннями.
- **Приховування внутрішніх помилок** — клієнт отримує лише загальне повідомлення про помилку, деталі логуються на сервері.
- Логування всіх критичних дій в `AuditLog` та `SystemAuditLog`.
- **MongoDB indexes** — оптимізовані індекси для швидких запитів (isDeleted+status, creator, department, tags, confidentiality, text index).
